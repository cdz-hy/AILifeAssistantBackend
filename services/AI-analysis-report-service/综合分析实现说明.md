# AI每日综合分析报告实现说明

## 概述

本文档详细说明了AI每日综合分析报告(`generateDailyReport`)的完整实现流程。该功能整合了**日程、财务、饮食、学习、天气**五个维度的数据,通过AI进行跨领域分析,生成包含综合效率评分、洞察建议和用户画像的智能报告。

## 核心设计理念

### 多模态数据交叉融合 (Cross-Domain Data Fusion)

不再单独分析各个模块,而是通过跨领域关联发现深层次的行为模式:

1. **学习 + 日程 = 拖延指数**: 如果任务截止日期临近但专注记录很晚才开始,说明拖延
2. **学习 + 财务 = 压力消费**: 如果学习完成度低,紧接着出现大额娱乐/购物支出,判定为压力型消费
3. **学习 + 饮食 = 能量供给**: 如果有高强度学习计划但饮食热量不足,给出能量不足预警
4. **日程 + 饮食 + 天气**: 根据日程强度、天气情况,建议合适的饮食调整

## 数据收集流程

### 1. 天气数据 (Weather Data)

**API**: `/schedule/api/weather/{cityCode}`

**收集内容**:
- 天气状况 (condition)
- 温度 (temp)
- 湿度 (humidity)

**用途**: 结合日程和饮食,给出环境适应性建议

### 2. 日程数据 (Schedule Data)

**API**: `/schedule/api/schedules/user/{userId}/range?startTime={start}&endTime={end}`

**收集内容**:
- 昨日睡眠时间 (最后一条日程的结束时间)
- 今日起床时间 (第一条日程的开始时间)
- 日程密度 (busy_level: low/medium/high)
- 重要事项列表 (important_events)

**分析逻辑**:
```java
if (totalSchedules >= 8) {
    busyLevel = "high";
} else if (totalSchedules >= 4) {
    busyLevel = "medium";
}
```

### 3. 财务数据 (Finance Data)

**API**: 
- `/finance/api/finance/transactions` (获取最近交易)
- `/finance/api/finance/transactions/stats/monthly?year={year}&month={month}` (月度统计)

**收集内容**:
- 今日支出总额 (daily_expense)
- 预算状态 (budget_status: normal/warning)
- 消费最多的分类 (top_category)

**分析逻辑**:
```java
// 判断预算状态
if (totalExpense > 5000) {
    budgetStatus = "warning";
}
```

### 4. 饮食数据 (Diet Data)

**API**:
- `/diet-record/api/diet-logs/{userId}/{date}` (饮食日志)
- `/diet-record/api/diet-logs/{userId}/{date}/stats` (营养统计)

**收集内容**:
- 热量状态 (calorie_status: low/normal/high)
- 进食规律性 (irregular_meal: true/false)
- 特殊备注 (notes: 如"缺少早餐"、"摄入咖啡因")

**分析逻辑**:
```java
if (totalCalories < 1200) {
    calorieStatus = "low";
} else if (totalCalories > 2500) {
    calorieStatus = "high";
}
```

### 5. 学习数据 (Study Data)

**API**:
- `/study/api/focus/user/{userId}/sessions` (专注会话)
- `/study/api/study-tasks/user/{userId}` (学习任务)

**收集内容**:
- 今日专注时长 (focus_minutes)
- 任务完成数 (tasks_completed)
- 任务总数 (tasks_total)
- 专注时间段 (focus_sessions: ["20:00-20:45"])

**分析逻辑**:
```java
// 统计今日截止的任务
if (dueDate.equals(date) || dueDate.isBefore(date)) {
    tasksTotal++;
    if (Boolean.TRUE.equals(isCompleted)) {
        tasksCompleted++;
    }
}
```

## AI提示词设计

### System Prompt (系统角色设定)

```
你是一位全能的"AI 个人生活首席参谋"。
你的任务是结合用户的【日程、财务、饮食、学习、天气】数据,进行跨领域的综合分析。

请完成以下三个任务:
1. **打分**:计算今日/昨日的"综合效率分"(0.0-10.0)。
2. **洞察**:给出一口气的"综合建议/洞察"(不超过 40 字)。
3. **画像更新**:根据最近的行为数据,重新评估用户的"AI 人格画像"。

【画像评估标准】:
- **核心效率时段**:根据学习模块的专注记录时间分布判断。
- **拖延指数**:根据学习任务的延期情况和日程的完成度判断 (0-1)。
- **消费习惯**:根据支出结构判断 (理性型/冲动型/保守型)。
- **饮食风险**:根据高糖高脂摄入及进食规律判断 (低/中/高)。
- **作息规律**:根据日程最早/最晚时间判断 (规律/熬夜/混乱)。

【分析逻辑】:
- **学习 + 日程 = 拖延指数**:如果任务截止日期临近但专注记录很晚才开始,说明拖延。
- **学习 + 财务 = 压力消费**:如果学习完成度低,紧接着出现大额娱乐/购物支出,判定为压力型消费。
- **学习 + 饮食 = 能量供给**:如果有高强度学习计划但饮食热量不足,给出能量不足预警。
- **日程 + 饮食 + 天气**:根据日程强度、天气情况,建议合适的饮食调整。

【综合效率分计算】:
Score = (日程完成率 * 0.3) + (专注时长达标率 * 0.3) + (预算健康度 * 0.2) + (饮食健康度 * 0.2)
```

### 1. 前端触发

### 2. 控制器处理

```java
@GetMapping("/daily/{userId}")
public ResponseEntity<GeneratedReport> getDailyReport(
        @PathVariable Long userId,
        @RequestParam String date) {
    LocalDate localDate = LocalDate.parse(date);
    GeneratedReport report = aiAnalysisService.getDailyReportByDate(userId, localDate);
    
    // 如果不存在或需要更新,则生成新报告
    if (report == null || needsUpdate(report)) {
        report = aiAnalysisService.generateDailyReport(userId, localDate);
    }
    
    return ResponseEntity.ok(report);
}
```

### 3. 服务层执行

```java
@Override
public GeneratedReport generateDailyReport(Long userId, LocalDate date) {
    // 1. 检查现有报告
    GeneratedReport existingReport = getDailyReportByDate(userId, date);
    
    // 2. 调用综合分析服务
    String analysisResult = comprehensiveAnalysisService
        .generateDailyComprehensiveReport(userId, date);
    
    // 3. 保存或更新报告
    GeneratedReport report = new GeneratedReport();
    report.setUserId(userId);
    report.setReportType("daily");
    report.setStartDate(date);
    report.setEndDate(date);
    report.setCreatedAt(LocalDateTime.now());
    report.setReportDataJson(analysisResult);
    
    if (existingReport != null) {
        report.setId(existingReport.getId());
        generatedReportMapper.update(report);
    } else {
        generatedReportMapper.insert(report);
    }
    
    return report;
}
```

### 4. 综合分析服务

```java
public String generateDailyComprehensiveReport(Long userId, LocalDate date) {
    // 1. 并行获取各模块数据
    Map<String, Object> aggregatedData = collectAllData(userId, date);
    
    // 2. 构建AI提示词
    String systemPrompt = getSystemPrompt();
    String userPrompt = objectMapper.writeValueAsString(aggregatedData);
    
    // 3. 调用大模型
    String aiResponse = llmClient.chat(systemPrompt, userPrompt);
    
    return aiResponse;
}
```

## 定时任务调度

### 调度配置 (AIAnalysisScheduler)

```java
@Scheduled(fixedRate = 2 * 60 * 60 * 1000) // 每2小时执行一次
public void scheduleDailyComprehensiveAnalysis() {
    LocalDateTime now = LocalDateTime.now();
    // 只在用户活跃时间段执行 (8:00 - 22:00)
    if (now.getHour() >= 8 && now.getHour() <= 22) {
        LocalDate today = LocalDate.now();
        for (Long userId : userIds) {
            jobScheduler.enqueue(() -> 
                aiAnalysisJobHandler.executeDailyComprehensiveAnalysis(userId, today)
            );
        }
    }
}
```

### 任务处理器 (AIAnalysisJobHandler)

```java
@Job(name = "执行每日综合分析")
public void executeDailyComprehensiveAnalysis(Long userId, LocalDate date) {
    System.out.println("开始执行用户 " + userId + " 在 " + date + " 的每日综合分析任务");
    
    // 1. 收集数据
    dataCollectionService.collectDailyData(userId, date);
    
    // 2. 生成报告
    aiAnalysisService.generateDailyReport(userId, date);
    
    System.out.println("每日综合分析任务执行完成");
}
```

## 性能优化

### 1. 缓存策略
- 每2小时强制更新一次报告
- 同一天内的查询优先返回缓存结果

### 2. 并行数据获取
```java
// 可以使用CompletableFuture实现并行获取
CompletableFuture<Map<String, Object>> weatherFuture = 
    CompletableFuture.supplyAsync(() -> getWeatherData());
CompletableFuture<Map<String, Object>> scheduleFuture = 
    CompletableFuture.supplyAsync(() -> getScheduleData(userId, date));
// ...
CompletableFuture.allOf(weatherFuture, scheduleFuture, ...).join();
```

### 3. 速率限制
LLMClient中已实现简单的速率限制,确保请求间隔至少1秒

## 错误处理

### 1. 数据获取失败
每个数据收集方法都有try-catch,失败时使用默认值

返回占位符JSON,确保系统可用性

### 3. 日志记录
关键步骤都有详细的日志输出,便于调试
